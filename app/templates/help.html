Getting Started

System Overview

This attendance system consists of three main components:

📱 Mobile App Flutter-based Android app for employees to clock in/out
→
🖥️ Middleware API FastAPI backend (this server) that processes punches and syncs data
→
🏢 ADMS Server ZKTeco attendance management server (existing infrastructure)

First-Time Setup Checklist

  • Configure ADMS Target — Go to Settings → ADMS Target and enter your ZKTeco server URL, serial number, and device name
  • Set ADMS Credentials — Go to Settings → ADMS Credentials and enter the username/password for ADMS sync
  • Create Punch Types — Go to Punch Types and add Clock In, Clock Out, Break Start, Break End, etc.
  • Add Branches — Go to Branches and add office locations with GPS coordinates and geofence radius
  • Create API Keys — Go to API Keys and generate keys for each employee device
  • Add Employees — Employees are auto-created when devices register, or you can pre-register them
  • Configure App Settings — Set minimum app version, sync intervals, and other global settings
  • Verify ADMS Sync — Check the ADMS Sync Dashboard to ensure data is flowing correctly

Architecture

System Flow:

Mobile App ──HTTPS──▶ Middleware API ──iClock Protocol──▶ ADMS Server
    │                      │
    │                      ├── PostgreSQL (Primary Database)
    │                      ├── Redis (Cache + Task Queue)
    │                      └── ARQ Worker (Background Sync)

Tech Stack:
  • Backend: Python FastAPI + SQLAlchemy + PostgreSQL
  • Mobile: Flutter + Drift (SQLite) + Riverpod
  • Cache: Redis 7 (via redis-py async)
  • Task Queue: ARQ (Redis-based async worker)
  • ADMS Protocol: ZKTeco iClock 501/502
  • Auth: API Keys (bcrypt-hashed) + JWT (admin UI)

Device Management

Device Registration Flow

  1. Employee installs app — Downloads and opens the Flutter mobile app
  2. Enters credentials — Server URL + API Key provided by admin
  3. Auto-registration — The app sends device info (UUID, model, name) to the server
  4. Registration appears in Dashboard — Status shows "Pending Approval"
  5. Admin approves — Click "Approve" on the pending device
  6. Assign branches — Select which office locations the device can punch from
  7. Device active — Status changes to "Active", employee can now punch

Device Statuses

Pending ApprovalDevice registered but not yet approved by admin
Pending BranchApproved but no branch assigned yet
ActiveFully operational — can punch attendance
SuspendedManually deactivated by admin — cannot punch

Managing Devices

  • Edit device — Change device name or reassign branches
  • Suspend device — Temporarily block a device (e.g., lost phone)
  • Delete device — Permanently remove a device registration
  • View logs — See all punches from a specific device
  • Bulk operations — Select multiple devices to approve or assign branches in bulk

Branch & Geofence Setup

Adding a Branch

  1. Go to Branches section in the admin sidebar
  2. Click "Add Branch" to open the form
  3. Enter branch name — e.g., "Jakarta Office", "Surabaya Warehouse"
  4. Set GPS coordinates — Use Google Maps to find the exact latitude/longitude of your office entrance
  5. Set radius — Geofence radius in meters (e.g., 100m for a typical office)
  6. Optional: Enable QR — Check "Enable QR Code Verification" if employees must scan a QR code at the entrance
  7. Save — The branch is now available for device assignment
Tip: Use a radius of 50-100m for single-building offices. Larger radius (200-500m) for campuses or factory compounds.

QR Code Verification

When QR verification is enabled for a branch:

  1. Generate a unique QR code data string (UUID recommended)
  2. Print the QR code and display it at the branch entrance
  3. Employees scan the QR code when punching
  4. The app verifies the scanned code matches the branch's expected data
  5. This provides proof of physical presence at the location

API Key Management

Creating API Keys

  1. Go to API Keys section
  2. Enter a label — Descriptive name (e.g., "John's Phone - Pixel 7")
  3. Set expiration — Optional: set key to expire after N days for security
  4. Create — The key is generated and shown ONCE
  5. Copy and share — Send the key securely to the employee
Important: API keys are shown only once at creation. If lost, you must generate a new key. Keys are stored as bcrypt hashes in the database.

Key Rotation

To rotate a key without disrupting the employee:

  1. Click the "Rotate" button next to the existing key
  2. A new key is generated immediately
  3. The old key remains valid for a 7-day grace period
  4. The employee can switch to the new key at their convenience
  5. After 7 days, the old key expires automatically

Security Monitoring

The API Keys table shows:

  • Last Used IP — The IP address that last used this key (detect anomalies)
  • Expiry Status — Green ✅ = more than 30 days, Yellow ⚠️ = less than 30 days, Red ⛔ = expired
  • Usage tracking — Last used timestamp is updated on every API call

ADMS Configuration

What is ADMS?

ADMS (Attendance Data Management System) is ZKTeco's server software for managing fingerprint/face attendance machines. This middleware bridges mobile app punches with your existing ADMS infrastructure, so employees can use their phones while attendance data still flows to your central ADMS server.

ADMS Target Setup

  1. Go to Settings → ADMS Target
  2. Server URL — The HTTP address of your ADMS server (e.g., http://192.168.1.100:8080)
  3. Serial Number — The device serial number registered in ADMS (this middleware pretends to be a physical device)
  4. Device Name — A descriptive name (e.g., "Middleware Bridge")
  5. Save — The system will attempt to connect

ADMS Credentials

  1. Go to Settings → ADMS Credentials
  2. Enter username and password for ADMS authentication
  3. Save — Credentials are encrypted at rest

How Sync Works

The middleware uses the ZKTeco iClock 501/502 protocol to communicate with ADMS:

  • Heartbeat — Sends a GET request every 60 seconds to keep the connection alive
  • Command parsing — Parses XML commands from ADMS (restart, sync time, etc.)
  • Push punches — Sends attendance records in ZKTransaction format
  • Acknowledgement — Confirms receipt of commands from ADMS

Sync Status Dashboard

Monitor sync health in the ADMS Sync Dashboard:

  • Stats cards — Total records, synced, pending, failed
  • Sync rate — Percentage of records successfully synced
  • Recent failures — Last 50 failed records with error messages and Retry button
  • Connection status — ADMS server up/down indicator
  • Worker status — ARQ task queue running/stopped indicator
  • Manual controls — Retry All Failed, Full Sync buttons

Retry Behavior

Failed syncs are retried automatically with exponential backoff:

  • First retry: 10 seconds
  • Second retry: 20 seconds
  • Third retry: 40 seconds
  • Fourth retry: 80 seconds
  • Fifth retry: 160 seconds (~2.5 minutes)
  • After 5 failures, the record remains as "failed" for manual retry
  • Failed records are automatically retried every 5 minutes via cron job

Punch Types

Configuring Punch Types

Punch types define what kind of attendance events employees can submit:

  1. Go to Punch Types section
  2. Click "Add Type"
  3. Code — A short identifier (e.g., "in", "out", "break_start")
  4. Label — Display name shown in the mobile app (e.g., "Clock In")
  5. Color — Hex color code for visual identification in the app
  6. Requires Geofence — Toggle whether this punch type needs GPS verification
  7. Save

Version Enforcement

The Min App Version setting allows you to force employees to update:

  1. Set the minimum version in App Settings
  2. When an outdated app connects, it receives the minimum version number
  3. The mobile app displays an "Update Required" screen
  4. The employee must update their app before they can punch
  5. This ensures all devices run a compatible version

Supervisor Management

Assigning Supervisors

  1. Go to Supervisors section
  2. Enter Supervisor ID — The employee ID of the supervisor
  3. Enter Team Member ID — The employee ID of the team member
  4. Assign — The mapping is saved
  5. Repeat for all team members under this supervisor

Supervisor Capabilities

Once assigned, supervisors can via the mobile app:

  • View team attendance — See real-time status of all team members (who's punched in, late, absent)
  • View history — Check detailed punch history for each team member
  • Review corrections — See pending attendance correction requests from their team
  • Approve/Reject corrections — Approve time adjustments or reject them with notes

Correction Requests

Employees can submit correction requests for:

  • Missing punch — Employee forgot to clock in/out
  • Wrong type — Punched "Break Start" instead of "Clock Out"
  • Wrong time — Device time was incorrect at time of punch

Supervisors review these requests and either approve (which updates the record) or reject with a reason.

Security Features

Defense in Depth

The system implements multiple layers of security:

1. API Key Authentication

  • All mobile API requests require a valid API key in the X-API-Key header
  • Keys are stored as bcrypt hashes (one-way, salted)
  • Keys can have expiration dates
  • Revoked keys are rejected immediately

2. Rate Limiting

  • Punch endpoints: 10 requests/minute per API key
  • Config endpoints: 30 requests/minute per API key
  • Prevents brute force and abuse

3. Punch Validation

  • Timestamp validation — Client time must be within 5 minutes of server time
  • Daily limit — Max 10 punches per employee per day
  • Duplicate detection — Same punch type within 5 minutes is rejected
  • Client ID idempotency — Each punch has a unique UUID to prevent duplicates on retry
  • Mock location flagging — GPS spoofing attempts are logged and visible to admins

4. Mobile Security

  • Certificate pinning — Prevents MITM attacks (configurable toggle)
  • Root/jailbreak detection — Blocks punching on compromised devices
  • Biometric authentication — Fingerprint/face required before punch
  • Biometric session timeout — Configurable re-authentication interval
  • Secure storage — API key stored in Android Keystore via flutter_secure_storage

Troubleshooting

Common Issues & Solutions

❓ ADMS server not connecting

  • Verify the server URL is correct and reachable from the middleware server
  • Check that the serial number matches what's registered in ADMS
  • Ensure the ADMS server is running and accepting connections
  • Check firewall rules — port 8080 (or your ADMS port) must be open

❓ Punches not syncing to ADMS

  • Check the ADMS Sync Dashboard for failed records and error messages
  • Verify ADMS credentials are correct in Settings
  • Try "Retry All Failed" button in the dashboard
  • Check the ARQ worker logs for detailed error information
  • Ensure the worker container is running: docker ps | grep worker

❓ Mobile app can't connect to server

  • Verify the server URL is correct in the app's Settings
  • Check that the backend server is running
  • Verify the API key is valid and not expired
  • Check that the API key has not been revoked in the admin dashboard
  • If using HTTPS, check certificate validity

❓ Device registration not appearing

  • Ask the employee to close and reopen the app to trigger re-registration
  • Check that the API key is correctly linked to the employee
  • Refresh the admin dashboard page
  • Check the server logs for registration attempts

❓ Database connection issues

  • Check that PostgreSQL container is running: docker ps | grep db
  • Verify database credentials in docker-compose.yml or .env
  • Check disk space: df -h — full disk can cause DB failures
  • Restart the database container: docker restart attendance_db

❓ Redis connection issues

  • Check that Redis container is running: docker ps | grep redis
  • Restart Redis: docker restart attendance_redis
  • Redis data is ephemeral — restarting is safe
